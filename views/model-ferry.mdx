The Model Ferry surfaces the model and provider layer of Hermes Agent activity. It shows which models handled which journeys, where the agent switched from one model to another, how many tokens each session consumed, and what those sessions cost when billing data was returned. Context compression events — which end a session and start a new context window — are also captured here because they often accompany model transitions.

## What the Model Ferry shows

The view draws on journey data from `GET /api/plugins/hermes-labyrinth/journeys` and displays two panels: recent model transitions across all journeys, and a summary of providers seen with their journey and token counts.

### Model sequence per journey

Every journey carries a `model_sequence` field — an ordered list of the models that handled that session. On most sessions this is a single-element list. It becomes a multi-element list when:

- The provider returned a different model identifier than the one configured.
- A model fallback occurred after repeated tool failures.
- Context compression ended the session and a subsequent session used a different model.

The model sequence is shown in the journey header as a left-to-right arrow sequence, for example:

```text
hermes-4-70b → claude-sonnet-4.5
```

### Context compression events

When a session's `end_reason` equals `compression`, the session ended because the context window was exhausted and the agent compressed its state to continue in a new session. This event appears on the Labyrinth Map as a `context_compression` crossing.

The Model Ferry treats compression-terminated sessions as significant because they often indicate the model reached its context limit and had to shed earlier conversation history. If you see a model transition immediately after a compression event in adjacent journeys, the two are likely related.

### Token counts

Each journey includes a `token_counts` object with the following fields:

| Field | Description |
|---|---|
| `input` | Tokens sent to the model in prompt and context. |
| `output` | Tokens generated by the model in the response. |
| `cache_read` | Tokens read from the provider's prompt cache (reduces cost on supporting providers). |
| `cache_write` | Tokens written to the provider's prompt cache. |
| `reasoning` | Tokens used for internal chain-of-thought reasoning on models that expose this (for example, extended thinking models). |

Token counts come directly from the provider API responses recorded by Hermes. Not all providers return every field — missing counts appear as zero.

### Cost estimates

<AccordionGroup>
  <Accordion title="estimated_cost">
    The cost estimate in USD computed by Hermes at session end, based on token counts and the model's configured pricing. May differ from the actual billed amount.
  </Accordion>
  <Accordion title="actual_cost">
    The actual cost in USD as reported by the provider, when the provider returns billing data in the API response. Not available on all providers.
  </Accordion>
  <Accordion title="cost_status">
    A status string indicating the reliability of the cost figures — for example, whether the estimate is based on cached token pricing or a live provider response.
  </Accordion>
</AccordionGroup>

## Recent transitions panel

The transitions panel lists model switches observed across recent journeys. Each entry shows:

- The previous model (or `—` for the first model in a journey).
- The new model the agent transitioned to.
- The cause of the transition as a human-readable string (for example, "2 tool failures · ctx 92%" or "subagent spawn").
- The journey ID the transition belongs to.
- The timestamp of the switch.

<Note>
  Transition causes are inferred from available session data and may not always be precise. A transition labeled "subagent spawn" reflects that a `subagent_spawn` crossing preceded the model change; it does not guarantee the spawn was the direct cause.
</Note>

## Providers seen panel

The providers panel summarizes every provider that appeared across recent journeys, showing the number of journeys handled and the approximate total token count. Provider status (`ok` or `unknown`) reflects whether the provider was reachable during those sessions.

## Notes on sparse model data

<Tip>
  On sessions where Hermes did not record a model — for example, sessions that ended abnormally before the model field was written — the `model_sequence` field is empty and the journey appears without a model label in the Journey Index. The Model Ferry will not show transitions for these sessions.
</Tip>

Model data reflects what Hermes has stored in `state.db`. If a session used multiple models through fallback logic but only one was recorded, the sequence will appear shorter than the actual number of transitions. Sessions from older Hermes versions that predate the `model_sequence` field will show a single-element or empty sequence.
