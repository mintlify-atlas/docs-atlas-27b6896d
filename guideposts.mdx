Guideposts are computed interpretations generated entirely from local Hermes state. They are not inferences about intent or predictions about future behavior — each guidepost is derived directly from the fields and crossings already recorded in `state.db`. They help you orient inside a journey quickly, spotting notable patterns (long durations, repeated failures, delegation boundaries) without requiring you to read every crossing manually. Guideposts never fabricate or estimate: if a field is absent from the session record, no guidepost that depends on it is emitted.

## Guidepost fields

Every guidepost object carries the following fields.

| Field | Type | Description |
|---|---|---|
| `kind` | string | Category of the observation. See the accordion below for all kinds. |
| `severity` | string | One of `info`, `notice`, or `warning`. Indicates how much attention the guidepost warrants. |
| `title` | string | Short, human-readable label for the guidepost. |
| `detail` | string | Optional additional context. Empty string when not applicable. |
| `evidence_refs` | array | List of references to the local data that produced this guidepost. Each ref carries a `kind` (`session`, `crossing`, or `tool`) and an identifying field or ID. |

### Severity levels

- **`info`** — Descriptive. No action required; the observation is useful for context.
- **`notice`** — Worth noting. The journey has a measurable characteristic (long duration, high tool use, approval boundaries) that may warrant a closer look.
- **`warning`** — Indicates a problem observed in the data, such as failed crossings or a repeated failing tool.

## Guidepost types

The following nine guidepost types are generated by `_guideposts_for()` in `dashboard/plugin_api.py`. Each accordion entry lists the kind, severity, what condition triggers it, and what the `evidence_refs` point to.

<AccordionGroup>
  <Accordion title="Model not recorded — missing_data / info">
    **Triggered when:** `model_sequence` is empty on the journey object. This means the Hermes session record did not capture a model name.

    **Evidence ref:** Points to the `model` field on the session record.

    ```json
    {
      "kind": "missing_data",
      "severity": "info",
      "title": "Model not recorded",
      "detail": "",
      "evidence_refs": [{"kind": "session", "id": "<journey_id>", "field": "model"}]
    }
    ```
  </Accordion>

  <Accordion title="Child journey — delegation / info">
    **Triggered when:** The journey has a `parent_journey_id` set — meaning it was spawned as a subagent by another session.

    **Evidence ref:** Points to the `parent_session_id` field on the session record.

    ```json
    {
      "kind": "delegation",
      "severity": "info",
      "title": "Child journey",
      "detail": "This journey has a parent session.",
      "evidence_refs": [{"kind": "session", "id": "<journey_id>", "field": "parent_session_id"}]
    }
    ```
  </Accordion>

  <Accordion title="Context compression occurred — context / info">
    **Triggered when:** The journey's `end_reason` is `"compression"`. The session ended because the context window was compressed rather than completing normally.

    **Evidence ref:** Points to the `end_reason` field on the session record.

    ```json
    {
      "kind": "context",
      "severity": "info",
      "title": "Context compression occurred",
      "detail": "",
      "evidence_refs": [{"kind": "session", "id": "<journey_id>", "field": "end_reason"}]
    }
    ```
  </Accordion>

  <Accordion title="Long journey — duration / notice">
    **Triggered when:** `duration_ms` exceeds 30 minutes (1,800,000 ms). The threshold is set by `LONG_JOURNEY_SECONDS = 30 * 60` in `plugin_api.py`.

    **Evidence ref:** Points to the `duration_ms` field on the session record.

    ```json
    {
      "kind": "duration",
      "severity": "notice",
      "title": "Long journey",
      "detail": "Duration exceeded 30 minutes.",
      "evidence_refs": [{"kind": "session", "id": "<journey_id>", "field": "duration_ms"}]
    }
    ```
  </Accordion>

  <Accordion title="High tool-call journey — tooling / notice">
    **Triggered when:** `tool_call_count` is 10 or more. Useful for identifying journeys with heavy tool use that may be worth auditing for efficiency or repeated calls.

    **Evidence ref:** Points to the `tool_call_count` field on the session record.

    ```json
    {
      "kind": "tooling",
      "severity": "notice",
      "title": "High tool-call journey",
      "detail": "",
      "evidence_refs": [{"kind": "session", "id": "<journey_id>", "field": "tool_call_count"}]
    }
    ```
  </Accordion>

  <Accordion title="N failed crossing(s) — failure / warning">
    **Triggered when:** One or more crossings in the journey have `status` set to `"failed"`. The title includes the exact count of failed crossings.

    **Evidence refs:** Up to 8 individual failed crossing IDs. Capped at 8 to keep the payload manageable for very noisy journeys.

    ```json
    {
      "kind": "failure",
      "severity": "warning",
      "title": "3 failed crossing(s)",
      "detail": "",
      "evidence_refs": [
        {"kind": "crossing", "id": "<crossing_id>"},
        {"kind": "crossing", "id": "<crossing_id>"}
      ]
    }
    ```
  </Accordion>

  <Accordion title="Repeated failing tool: {tool} — loop / warning">
    **Triggered when:** The same tool name appears in more than one failed `tool_result` crossing. One guidepost is emitted per tool that meets this condition. The title includes the tool name.

    **Evidence ref:** Points to the tool name via a `tool` kind reference. The `detail` field states how many failed results were observed.

    ```json
    {
      "kind": "loop",
      "severity": "warning",
      "title": "Repeated failing tool: run_bash",
      "detail": "3 failed results were observed.",
      "evidence_refs": [{"kind": "tool", "name": "run_bash"}]
    }
    ```
  </Accordion>

  <Accordion title="Delegation boundary crossed — delegation / info">
    **Triggered when:** At least one crossing in the journey has `type` set to `"subagent_spawn"`. This means the agent delegated work to a child agent during the journey.

    **Evidence refs:** All `subagent_spawn` crossing IDs in the journey.

    ```json
    {
      "kind": "delegation",
      "severity": "info",
      "title": "Delegation boundary crossed",
      "detail": "",
      "evidence_refs": [{"kind": "crossing", "id": "<crossing_id>"}]
    }
    ```
  </Accordion>

  <Accordion title="Approval boundary crossed — approval / notice">
    **Triggered when:** At least one crossing in the journey has `type` set to `"approval"`. This means the agent reached a decision point requiring explicit user or policy approval during the journey.

    **Evidence refs:** All `approval` crossing IDs in the journey.

    ```json
    {
      "kind": "approval",
      "severity": "notice",
      "title": "Approval boundary crossed",
      "detail": "",
      "evidence_refs": [{"kind": "crossing", "id": "<crossing_id>"}]
    }
    ```
  </Accordion>
</AccordionGroup>

## Where guideposts appear

Guideposts are computed in two contexts.

**Per-journey — Crossing Inspector**

When you open a journey in the Labyrinth tab and navigate to the **Crossing Inspector**, guideposts for that journey are listed at the top of the panel. Each guidepost links to its evidence crossings or session fields so you can navigate directly to the relevant data.

Guideposts are also included in the `guideposts` array returned by the journey detail endpoint (`GET /api/plugins/hermes-labyrinth/journeys/{journey_id}`) and embedded in both the Markdown and JSON export formats.

**Global — recent journeys endpoint**

The `/api/plugins/hermes-labyrinth/guideposts` endpoint aggregates guideposts across the most recent journeys (up to the configured `limit`, defaulting to 10). Each guidepost in the global list carries a `journey_id` so you can trace it back to the originating session. This is the data source for the global Guideposts summary in the dashboard.

<Note>
  Every guidepost is fully explainable from local Hermes state. The `evidence_refs` on each guidepost point directly to the session field, crossing ID, or tool name that produced it. No external data is consulted and no values are fabricated. If the evidence does not exist in `state.db`, the guidepost is not emitted.
</Note>
